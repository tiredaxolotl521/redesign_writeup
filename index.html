Overview

The goal of this assignment to practice the workflow of redesigning a simple website. I analyzed and identified flaws in an existing interface, create low-fidelity and high-fidelity prototypes for various screen sizes, and build a responsive website based on those prototypes.

Part 1: Identifying Usability Problems (arXiv website)

I chose the arXiv website because it is a commonly used website where a lot of researchers use it to post both pre-published and published papers. Because researchers are notorious for being utilitarian, the design and layout of this website lacks a good user interface. Futhermore, this design is the original design that was used in 1980s. Thus, it needs a good makeover. Below are images of the website and the WAVE report.

Click to vist arxiv.org

Finding Problems

  • Usability
    • Scientific papers are presented in a traditional hierarchy
    • Have to step through several pages to reach the papers you want (topic → subtopic → year → month → papers)
    • Lot of pages to sift through before landing on your desired search
    • Contains a search button hidden in the upper right hand size (very small)
    • Search button has ablity to serach with various subcategories
    • Has full directory stack so you can trace back your search
    • Main display presented you with every topic and subtopic (overwhelming)
  • Learnability
    • Easy transferability of knowledge from searching through folders from a computer
    • Easy ability to know how to search and build a conceptual model, but difficult execute to find a specific article because there are a lot of filter steps
    • Low learning curve, hard to master
  • Memorability
    • Three distint colors: maroon red, black and white
    • Confusing name (good that it gets people to recognize that it hard to pronounce, but bad that no one knows how to stay the website name)
    • Not intuitive that the X is chi arXiv pronounced archive
    • Topics laid out in a list are all hyperlinks to another page
    • Nothing outstanding other than the large red banner on the top and then white page full of hyperlinks
    • Memorable title logo (yellow smilly face with a cross on a green background)
    • Basic 1980s website design with minimal frills
  • Conceptual model
    • Viewed as a database to look up scientific papers
    • The homepage suggests that the best way to look up papers is to have a topic in mind and a time frame of papers you want to search through

Accessibility

  • There are 3 errors, which consist of an empty link and 2 broken ARIA items. The fact that a link is empty may be misleading and a broken ARIA item prevents web applications from being accessible to people with disabilities. I would have never known of these errors if I did not use WAVE.
  • There are 3 alerts: 1) skipped heading level, 2) a redundant link and a 3) noscript element. These alerts do not impact the user in any major way since they do not alter the way the user interacts with the interface.
  • There are 35 structural elements and 9 features which make for an interesting, yet simple web interface. It has a decent complexity as well as enough simplicity to make the user understand what is going on.
  • There are 244 ARIAs on this single page, 238 of which are ARIA labels. These 238 labels are all displayed to a user all at once. Thus, it can be overwhelming to the user and lead to bad accessibility.
  • There are no contrast errors so the website uses a great color scheme and all the font sizes are accessible.
  • No way to have search results show up based on preference (date, author, number of times cited, etc.)
  • No quick way to gather information from results (ie. if you only want to see how many related papers there are, you have to step through severl pages to get there, no quick access button)
  • Hard to know which filters someone is using
  • Lots of text overwhelming the user all at once
